1. Introduction

Geraci CRM (a Twenty fork) bundles a complete marketing toolkit alongside the core CRM. This manual is the full reference for everyone who runs marketing — set up campaigns, build sequences, capture leads with forms, design landing pages, and read the analytics that tell you whether any of it worked.

The CRM is at crm.geracillp.com. Sign in with your workspace email; if you don't have an account yet, ask the workspace owner to invite you (Settings → Members).

What changed since the last manual: Everything navigation is new. The marketing area now has a contextual sidebar (filter, pinned, recent, visit history), a global Cmd+K palette, dozens of keyboard shortcuts, and a HubSpot-style visual landing page builder. If you used the older version, sections 2-4 below are the most important read.

2. Navigation & the contextual sidebar

2.1 Where things live

The left side of the screen has three panels (left to right):

  1. Workspace drawer — the dark navy rail furthest left. This is the main app navigation: Workspace, Marketing, Other (Settings, Documentation).
  2. Marketing Tool Sidebar — the second column, contextual to whatever marketing section you're in. Shows a section title, + New button, filter, Pinned, Last visited, Frequently visited, and Recent records.
  3. Main work area — record list or record show page, depending on where you are.

The Marketing drawer (in the dark rail) groups all marketing surfaces in one place:

  • Email Campaigns (g 1)
  • Marketing Campaigns (g 2)
  • Sequences (g 3)
  • Forms (g 4)
  • Analytics (g 5)
  • Audiences (People) (g 6)
  • Landing Pages (g 7)

Each item shows a live record count next to it. The little number after  ·  is the total. g followed by the digit is the keyboard shortcut that navigates there from anywhere.

2.2 The Marketing Tool Sidebar (second column)

This is the workhorse. Whenever you're on a marketing section — Email Campaigns, Sequences, Forms, Landing Pages, etc. — the sidebar shows up populated with content for that section.

2.2.1 Header

  • An icon chip (Mail, Send, etc.) — clickable to collapse the sidebar to a 24px strip.
  • The section title (e.g. "Email Campaigns") with a count chip showing the total record count.
  • If you're on a record show page, a star icon (☆ or ★) lets you pin / unpin this page with one click. Or press p.
  • A ? button opens the keyboard shortcut overlay.
  • Subtitle is a quick description on index pages, but turns into a breadcrumb (Section › Record name) when you're viewing a record.

2.2.2 + New type

Below the header. Click it and a small inline form appears with a name input and Cancel / Create buttons. Type a name, hit Enter, and the new record opens. Esc cancels. The shortcut c opens this form from anywhere on the route.

2.2.3 Filter

Type to narrow Pinned, Recent, Last visited, and Across-sections. Press / from anywhere on a marketing route to jump into the filter without using the mouse. Esc clears the filter; press it again to unfocus. The filter is remembered per section — leave Email Campaigns filtered to "Q4", come back later, the filter is still there.

Type at least 2 characters to also search across other marketing sections — the filter shows matching campaigns / marketing campaigns / sequences / forms / landing pages from all sections, not just the one you're on.

Recent searches show as quick-select chips when the input is empty — click a chip to re-run that search. × on a chip removes it; the rightmost Clear chip wipes the whole list.

While the filter is active, press ↑ / ↓ to walk a highlighted cursor through results. Enter opens the highlighted record. Tab moves focus from the input to the first row.

2.2.4 Section

One row: "All type" — links to the section's full table view. The badge with the section's record count is shown nearby in the title.

2.2.5 Pinned

Records you want one-click access to. Click the ☆ on any Recent row to pin it; click the ★ on a Pinned row to unpin. New pins go to the top of the list.

  • Position numbers 1–9 show as small chips on the first nine pinned records. Press 1–9 from anywhere on the route to jump to that pinned record.
  • Drag to reorder — grab the ⋮⋮ handle and drop a row above another to change its order.
  • Double-click to rename — double-click any pinned row's label to give it a sidebar-only alias. Press Enter to save, Esc to cancel. Aliases are local to the sidebar; the underlying record name doesn't change.
  • Right-click → Move to top promotes a pinned record to position 1.
  • Right-click → Unpin removes it.
  • Right-click → Open in new tab / Copy link work on any sidebar row, not just pinned.
  • Clear in the section header unpins everything.
  • n / b walk to the next / previous pinned record.

2.2.6 Last visited

The last 8 record show pages you opened, across all marketing types. Each row has a section badge ("Email", "Seq", "Form", "Page", etc.) and a relative-time chip (5m, 2h, yest, 3d, Apr 24). If you've visited a record more than once, it shows 5× 2h with the visit count.

Clear in the header wipes visit history. Right-click any row → Remove from history to drop just that one. Visits older than 30 days auto-prune on read.

2.2.7 Frequently visited

Records you've visited 3+ times in the current section that you haven't pinned. Surfaced at the top of the sidebar so you can see what you keep coming back to and decide whether to pin it. Each row shows the visit count (e.g. 5×).

2.2.8 Recent

The 5 most recently created records in the current section. Each row has a ☆ to pin. If the section has more records than fit, a "→ View all N in table" link at the bottom takes you to the index view. The ↻ button refreshes.

2.2.9 Across sections (search results)

Only shows when the filter is active and there are matching records in other marketing sections. Each result has the type badge + record name and links straight to its show page.

2.3 Resizing & collapsing the sidebar

  • Drag to resize — pointer-down on the right edge of the sidebar and drag. Width clamps to 200–480px and persists across reloads. Double-click the resize handle to reset to 240px.
  • Collapse to a 24px strip — click the chevron pinned to the right edge, click the icon chip in the header, or press [. When collapsed the section title is shown in a vertical "writing-mode" hint; click the hint or the chevron to expand.
  • Auto-collapse — the sidebar auto-collapses if your window is narrower than 1100px. A toast confirms.

2.4 Right-click context menu

Right-click any sidebar row (Pinned, Recent, Last visited, Across sections, Frequently visited) for:

  • Open in new tab — opens the record in a new browser tab.
  • Copy link — writes the absolute URL of the record to your clipboard. A toast confirms.
  • Pin / Unpin — only on Pinned + Recent rows.
  • Move to top — only on Pinned rows. Moves the record to position 1 in the pinned list.
  • Remove from history — only on Last visited rows.

3. The Quick Switcher (Cmd+K)

Press ⌘+K (Mac) or Ctrl+K (Windows / Linux) on any page in the CRM — settings, records, dashboards — to open a centered command-palette overlay.

The switcher is the fastest way to navigate, especially when you're not on a marketing route and the contextual sidebar isn't available.

3.1 What's in the switcher

When you first open it (empty input):

  • Recent searches — quick-select chips for queries you've typed before.
  • Today — records you visited in the last 24 hours.
  • Recently visited — broader visit history (last 8, deduped against Today).
  • Sections — all 7 marketing sections plus Settings, with their g+digit shortcut hints.

When you start typing (≥ 2 characters):

  • Actions — verbs like "New Email Campaign", "New Landing Page", "Toggle sidebar collapse", "Open Settings".
  • Records — live ilike search across campaigns / marketing campaigns / sequences / forms / landing pages.

Records that are pinned in the marketing sidebar get a small ★ next to their badge in the switcher results, so you can eyeball familiar records at a glance.

The footer shows hints (↑↓ nav · Tab group · ↵ open · ⇧↵ tab · ⌘P pin · + new · ⌫ close) and the live result count.

5. Email Campaigns

Email Campaigns are one-off and scheduled marketing emails. Each campaign has a name, subject, body (built visually), recipients, and a status (Draft → Scheduled → Sending → Sent / Paused).

5.1 Creating a campaign

  1. Press g 1 or click Marketing → Email Campaigns in the drawer.
  2. Click + New email campaign in the sidebar (or press c).
  3. Type a name (e.g. "Q4 Newsletter") and press Enter. The new campaign opens.
  4. Fill in Subject, From email, From name in the right-rail.
  5. Build the body in the Email builder tab.
  6. Add recipients — either bulk via "Add recipients from view" or one-at-a-time.
  7. Send a test (Send test) to confirm the rendering looks right.
  8. Either Send now or set a Scheduled Date for delivery later.

5.2 The email builder

The body is composed of sections, each split into columns, each holding modules:

  • Heading — H1 / H2 / H3 with alignment, color, weight controls.
  • Text — rich-text block with token support ({{contact.firstName}}, etc.).
  • Button — call-to-action with bg / text colors, padding, radius.
  • Image — uploads through the Twenty file system.
  • Divider · Spacer · HTML — layout primitives + raw HTML escape hatch.
  • Social · Footer — social icon row + footer with literal {{unsubscribe_link}} token.

Section layouts: 1 column, 2 columns, 3 columns, 4 columns, or asymmetric 1/2 and 2/1. Drag the section divider to change widths. Modules drag-reorder within a column.

5.3 Tokens and substitutions

Anywhere a text or button accepts content, you can use these tokens. The send executor replaces them at delivery time:

  • {{contact.firstName}} · {{contact.lastName}}
  • {{contact.email}}
  • {{contact.companyName}}
  • {{unsubscribe_link}} — required in every send (compliance)

5.4 Test sends & preview links

Before sending, use:

  • Send test — sends the current body to one or more email addresses you specify, with a sample contact's substitutions.
  • Generate preview link — produces a 7-day signed URL where anyone can view the rendered email in a browser. Useful for sharing with stakeholders for review.

5.5 Stats

Once sent, the campaign tracks recipientCount, sentCount, openCount, clickCount, bounceCount, unsubscribeCount. The Marketing Analytics page (section 11) aggregates these across all campaigns.

6. Marketing Campaigns

Marketing Campaigns are groupings — they tie together emails, sequences, forms, and landing pages that all serve a single initiative (e.g. "Spring 2026 Product Launch"). They give you one place to see everything related to a campaign, plus aggregated metrics.

6.1 Creating one

  1. g 2 or Marketing → Marketing Campaigns.
  2. + New marketing campaign, name it, hit Enter.
  3. On the new record, set Status, Owner, Start date, End date, Goal.

6.2 Linking emails to a campaign

Two ways:

  • Primary FK — set Marketing Campaign on an email's record-show page.
  • Additional campaigns (many-to-many) — use the Additional Marketing Campaigns field to link an email to any number of other marketing campaigns. Useful when one email is shared across two initiatives.

7. Sequences

Sequences are multi-step automated cadences — drip campaigns. Each step is a delay + an action (send email, wait, branch on engagement, etc.).

7.1 Building a sequence

  1. g 3 or Marketing → Sequences.
  2. + New sequence, name it, open the record.
  3. In the Cadence tab, add steps:
    • Send email — pick a campaign template.
    • Wait — N days / hours.
    • Branch — split if the recipient opened or clicked.
  4. Set the Entry trigger: form submission, list membership, manual enroll.
  5. Activate by setting Status to Active.
Heads up: Once active, the sequence runs on its own. Test with a small audience first by using a tag or test list as the entry trigger.

8. Forms

Forms capture leads from external pages or embed-style placements. Each form publishes a public URL (https://crm.geracillp.com/forms/<form-id>), an iframe embed snippet, and a JSON submission API for posting from any external domain.

8.1 Creating a form

  1. g 4 or Marketing → Forms.
  2. + New form, name it.
  3. Add fields in the Builder tab. The Type dropdown supports 16 field types: short text, long text, number, boolean (checkbox), rating (1-5 stars), date, date & time, email (validated), phone (E.164), URL (validated), address, full name, file upload, single-select, multi-select, and record (CRM lookup).
  4. Configure:
    • Submit button label
    • Success message (or redirect URL)
    • Auto-create person — if on, every submission creates a new Person record.
    • Tags to apply on submission — string array. When auto-create is on, these tags are merged into the new (or matched) Person's tags field. Use them to segment "submitted Newsletter signup form" vs "submitted Demo request form".
  5. Publish — flips status to PUBLISHED and exposes the public URL.

8.2 Embedding (iframe)

From the form's record show page, copy the iframe snippet from Public URL → Embed code. Paste it into any HTML page. The iframe loads the CRM-rendered form and posts back to the CRM directly — no JS required on the host page. This is the simplest integration path; the form's UI is the CRM's UI.

8.3 Custom-form integration (JSON API)

If you want to build your own form on a different domain (your marketing site, a custom landing page, etc.) and post to the CRM, use the public submission endpoint directly. No auth, no API key, just a POST.

Endpoint

POST https://crm.geracillp.com/forms/<workspace-id>/<form-id>/submit
Content-Type: application/json

{
  "fields": {
    "email": "user@example.com",
    "firstName": "Jane",
    "lastName": "Doe",
    "phone": "+15551234567",
    "company": "Acme Inc."
  },
  "submitterEmail": "user@example.com",
  "submitterName": "Jane Doe"
}

workspace-id is a stable UUID for the workspace (same one you see in admin URLs). form-id is the form record's id — visible at the top of the form's record show page or via the embed iframe URL.

Body shape:

  • fields — required. Map of { formFieldName: value }. Field names match the Name set on each field in the Builder tab (default is the field's label, lowerCamelCased). Values are typed loosely; the server casts to the field's declared type.
  • submitterEmail — optional. If present, used for dedup against existing People records and as the recipient for the confirmation email (when Send confirmation email is on).
  • submitterName — optional. Used as firstName if not otherwise provided in fields.
  • captchaToken — required when the form has Bot protection enabled (see 8.3.1). Cloudflare Turnstile token from the form-host page's widget.
  • formLoadToken — required when the form has Require form-load token enabled. Signed JWT issued by the GET /schema endpoint; the host page passes it through verbatim on submit.

Response (200 OK):

{
  "submissionId": "<uuid>",
  "personId": "<uuid|null>"
}

If Auto-create person is on, personId is the id of the created (or matched, by email) Person record. If off, personId is null.

Error responses:

  • 400 — validation failure (missing required field, pattern mismatch, etc.). Body: { "errors": { "fieldName": "human message" } }.
  • 404 — form not found, not PUBLISHED, or workspace not found.
  • 500 — server error. Lead auto-creation failures are logged but don't fail the submission — the FormSubmission record is still saved.

CORS

The endpoint is configured with origin: true globally — the response's Access-Control-Allow-Origin header echoes the request Origin, so cross-origin POSTs from any external domain work in browsers. Access-Control-Allow-Credentials: true is also set (for the cookie-credentialed admin app), but you don't need to pass cookies for form submissions.

Custom-form fetch example (browser-side JS)

const form = document.querySelector('#my-form');

form.addEventListener('submit', async (e) => {
  e.preventDefault();
  const data = new FormData(form);

  const res = await fetch(
    'https://crm.geracillp.com/forms/<workspace-id>/<form-id>/submit',
    {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        fields: Object.fromEntries(data),
        submitterEmail: data.get('email'),
        submitterName: data.get('firstName'),
      }),
    },
  );

  if (res.ok) {
    const { submissionId, personId } = await res.json();
    // redirect, show success message, fire analytics, etc.
  } else {
    const { errors } = await res.json();
    // surface field-level errors to the user
  }
});

Server-to-server submissions

Same endpoint works from a backend (Node, PHP, Python, etc.) — no CORS to worry about server-side. Useful for piping leads from a third-party form builder (Typeform, JotForm, Google Forms via webhook) into the CRM.

8.3.1 Bot protection (built-in)

The CRM has six native bot-protection layers configurable per-form. Stack as many as fit your threat model. All are validated server-side at the submit endpoint, in this order (cheap → expensive):

LayerForm field(s)What it stopsFailure response
Origin allowlist Allowed origins (string array) Cross-site spam (someone steals your form ID + posts from a different domain) 403 Forbidden
Per-IP rate limit Rate limit (per IP per minute) (number) Rapid-fire bots posting in tight loops 429 Too Many Requests
Disposable email blacklist Reject disposable emails (toggle) Mailinator / temp-mail signups for spam 400 Bad Request
Form-load token (JWT) Require form-load token, Minimum submit time Bots that scrape the form ID and POST without ever rendering the form. Time-trap also catches instant-submit replays. 403 Forbidden
Honeypot field Honeypot field name Form-fillers that auto-populate every field 200 OK with "honeypot-rejected" (silent drop)
Cloudflare Turnstile Bot protection, Turnstile site key Sophisticated bots that render the page + behave like humans 403 Forbidden

Heads up on what's an account vs. what's not: only Cloudflare Turnstile (the bottom row) is a third-party service that needs a sign-up + API keys. Everything else — honeypot, origin allowlist, rate limit, disposable email check, form-load token — is native to the CRM. You toggle them on in the form's Settings tab and they Just Work. No accounts, no keys, no dashboards anywhere. Don't go looking for "honeypot.io" or "rate-limit.com" — they don't exist; honeypot is a 25-year-old web technique, not a vendor.

Origin allowlist

Set Allowed origins on the form to ["https://customer-site.com", "https://staging.customer-site.com"] (or just hostnames like "customer-site.com"). The server rejects POSTs whose Origin or Referer header doesn't match. Empty array = all origins allowed.

Per-IP rate limit

Set Rate limit to 5 for a 5-submissions-per-60s ceiling per source IP. The counter is in-memory (rolling window). 0 / null disables. Returns 429 when exceeded.

Disposable email blacklist

Toggle Reject disposable emails. The server checks both body.submitterEmail and body.fields.email against a baked-in list of common temp-mail domains (Mailinator, Tempmail, 10minutemail, Guerrilla Mail, Yopmail, etc.). Returns 400 with { errors: { email: "Please use a non-disposable email address" } } on hit.

Form-load token (JWT) + time-trap

Toggle Require form-load token. The flow:

  1. Browser GETs /forms/<wid>/<fid>/schema — response includes formLoadToken (a 24h-lifetime signed JWT).
  2. Browser POSTs /forms/<wid>/<fid>/submit with body.formLoadToken set to that token.
  3. Server verifies signature, subject, claims, expiration, and age against Minimum submit time (seconds) (default 2). Submissions with a token age below the floor are rejected as instant-fire bots.

The /schema endpoint always issues a formLoadToken regardless of whether the form requires it — costs nothing and lets you flip the requirement on later without code changes on the host page.

Honeypot field

Honeypot is not a service — no account, no API key, no third party. Just a hidden form field that bots fill in and humans don't.

  1. In the CRM, set Honeypot field name on the form to any string. Common choices: company_url, website, fax (deliberately unsuspicious).
  2. On your custom form, add an input with that exact name, positioned off-screen with CSS: 
    <input
  type="text"
  name="company_url"
  tabindex="-1"
  autocomplete="off"
  style="position:absolute;left:-9999px;width:1px;height:1px;opacity:0;"
/>
  3. POST the form normally — the honeypot value will be empty for real users: 
    {
  "fields": {
    "email": "user@example.com",
    "firstName": "Jane",
    "company_url": ""
  }
}

Real visitors never see the field (off-screen, no tab stop, autocomplete disabled). Bots that auto-fill every input on the page populate it. When the server sees a non-empty honeypot, it returns 200 OK with { "submissionId": "honeypot-rejected", "personId": null } and silently discards the submission — no FormSubmission record persisted, no notification email sent, no Person created. The bot's logs say "success" and it never learns why it didn't actually convert.

Why off-screen and not display:none? Some bots specifically skip display:none fields to dodge naive honeypots. Off-screen positioning + tabindex=-1 + autocomplete=off is the most defeat-resistant version.

Cloudflare Turnstile

Toggle Bot protection and set Turnstile site key on the form. Add CLOUDFLARE_TURNSTILE_SECRET_KEY to the server env. Embed Turnstile on your form page (script tag from Cloudflare); when the visitor solves the challenge, the widget produces a token. POST it in body.captchaToken. Server verifies via Cloudflare's siteverify before any field validation. Failure returns 403.

Recommended stack (zero third-party accounts)

For a low-effort baseline that stops ~99% of bot traffic without signing up for anything:

  1. Honeypot field name: company_url (and add the off-screen input shown above to your form)
  2. Require form-load token: on, with Minimum submit time = 2
  3. Rate limit: 5
  4. Allowed origins: ["https://your-domain.com"]

None of those four require an external account, API key, or dashboard — they're all just toggles in the CRM Settings tab plus a few lines of HTML/JS on your form. For high-volume consumer forms (newsletter signup with active spam campaigns), add Turnstile on top — that's the only one that needs a Cloudflare account.

8.3.2 Rate limiting (edge layer)

The per-form Rate limit field above runs in-app and is per-(form, IP). For workspace-wide spike protection (someone pounding /forms/*/submit with rotating form IDs), add a Cloudflare WAF rate-limit rule on the path itself — typically 20 req / 60s per IP. The two layers are complementary: in-app limits per-form abuse, edge limits aggregate flooding.

8.4 Submissions

Each submission shows up under the form's Submissions tab. If Auto-create person was on, the corresponding Person record is linked. The Person inherits any tags configured in Tags to apply on submission on the Form. If the Person already existed (matched by email), the new tags are unioned onto the existing tags array — you don't lose tags from earlier form submissions.

9. Landing Pages

Landing Pages are public web pages built inside the CRM. Each page has a slug (the URL path), title, status (Draft / Published / Archived), meta title, meta description, and a visual design built from sections + columns + modules — same mental model as the email builder.

9.1 Creating a landing page

  1. g 7 or Marketing → Landing Pages.
  2. + New landing page, give it a title, hit Enter.
  3. The new record opens with three tabs: Builder, Preview, Settings.
  4. In Settings, set the slug (e.g. spring-launch) and meta tags.
  5. In Builder, design the page (next subsection).
  6. Set Status to PUBLISHED. The page goes live at /lp/<slug>.

9.2 The builder

Three panels:

Left rail — module library

Click any tile to add a module to the currently-selected section. If no section exists yet, clicking auto-creates one. Modules are grouped:

  • Layouts — 1 / 2 / 3 / 4 columns, plus 1/2 and 2/1 asymmetric splits. Click a layout tile to add a new section with that layout.
  • Layout modules — Header / Nav (top bar with brand + links + CTA), Hero (full-width banner + heading + subheading + CTA), Feature list (2/3-col grid or list of icon + title + body cards), Social (icon row).
  • Content — Heading (H1–H4), Text (HTML body), Button (CTA).
  • Media — Image (with width + alignment + optional click-through link), HTML (raw escape hatch).
  • Forms — Form embed (drops in a published form from the Forms section by ID).
  • Spacers — Divider (line), Spacer (vertical gap).

Center — live canvas

Each section renders live with its modules. Click a section to select it (orange dashed border). Click a module to open the editor (right rail). Sections show:

  • ↑ / ↓ — move section up / down in the page
  • × — delete section
  • A chip strip below the section's modules — click any chip to add another module to this section

The top bar shows the active section's layout picker (1 / 2 / 3 / 4 / 1-2 / 2-1) and a + Add section button.

Right rail — module editor

Opens when you click a module. Type-specific fields:

  • Heading / Text / Button — content + alignment + colors + size.
  • Image — URL + alt + width % + alignment + optional link.
  • Hero — heading, subheading, CTA label / link, bg color, optional bg image URL (with dark overlay), text color, alignment, min-height.
  • Feature list — layout (2-col / 3-col / list) + colors. Add / edit / remove items inline (icon + title + body each).
  • Form embed — paste a form's UUID. Optional heading + CTA label overrides.
  • Header / Nav — brand text + logo URL, link list (label + href), CTA, bg / text colors.
  • Social — icon size + spacing + alignment + per-platform links.

The × in the editor's header closes it.

9.3 Preview & publish

The Preview tab renders the page as visitors will see it (no editor chrome). When status is PUBLISHED a Live at: url bar appears at the top of the widget — click the URL to open the public page in a new tab.

The public URL pattern is https://crm.geracillp.com/lp/<slug>. Each visit increments the page's viewCount (visible on the record).

9.4 Migration note

Pages created before the visual builder shipped used a simpler section format. They auto-migrate to the new design schema on first open — no action needed. Old pages keep rendering at their public URLs throughout the migration.

10. Audiences (People)

People are your contact records — leads, customers, partners. Same record shape as the core CRM's Person object; the marketing tools layer adds segmentation features on top.

10.1 Adding people

  • Manually — g 6 → + New person. Fill in fields.
  • Form submissions — when Auto-create person is on for a form (section 8.1), each submission creates a Person.
  • Bulk import — see Admin section 12.2.
  • HubSpot migration — there's an in-progress scripted import via the metadata API. Companies + people are imported; deals + notes are pending. Talk to admin if you need to refresh from HubSpot.

10.2 Segmentation

Use Twenty's standard list / tag / view machinery. Saved views in the People section show up as Views in the Marketing Tool Sidebar — one click to load that filter on the index page.

11. Marketing Analytics

g 5 or Marketing → Analytics. The page aggregates send / open / click / bounce / unsubscribe stats across all email campaigns, plus per-marketing-campaign rollups.

Charts:

  • Sends over time — week / month series.
  • Engagement funnel — sent → delivered → opened → clicked.
  • Top campaigns — by open rate, click rate, conversion.
  • Per-marketing-campaign rollup — aggregates all child emails / sequences / forms / landing pages of each marketing campaign.

The Marketing Tool Sidebar on this page also surfaces a Recently updated cross-section feed — the 8 most-recently-touched marketing records of any type, so you can dive into whatever you were last working on.

12. Admin / housekeeping

12.1 Workspace settings

⌘ , or Other → Settings. The bits relevant to marketing:

  • Email sender domain — DKIM / SPF setup. Without proper auth, sends will land in spam.
  • Members — invite / revoke users.
  • API keys — for the metadata-API import scripts.

12.2 Bulk import gotchas

If you're loading contacts at scale via the API, watch out for:

  • Rate limits — the metadata API throttles. Use batches of ~100 with retry-on-429.
  • E.164 phone format — phones must include country code (+15551234567). Plain US numbers fail validation.
  • Reserved field names — type, status have special meanings; check field docs before mapping.
  • Split-and-retry — if a batch fails, halve it and retry recursively until you find the bad row.

12.3 Deployment notes (for admins)

The CRM runs on Railway pulling ghcr.io/geraci-llp/twenty-crm:railway. Each push to main triggers a GitHub Actions image build (~10 min) and a Railway redeploy (~10 min). Total ~20 minutes from push to live.

If a feature requires a new database column or workspace entity, the admin must run workspace:sync-metadata against the live database after the deploy. Check the deploy notes for any migration step.

13. Troubleshooting

13.1 "I don't see the sidebar"

The Marketing Tool Sidebar only renders on marketing routes — /objects/campaigns, /objects/sequences, /marketing/analytics, etc. If you're on a non-marketing page (like Settings), it won't appear. Use ⌘ K instead.

13.2 "Cmd+K doesn't open"

Make sure your cursor isn't focused inside an editor that grabs ⌘ K for itself (some rich-text editors do). Click somewhere outside, then try again. If it still doesn't open, hard-refresh (⌘ ⇧ R) — the latest deploy may not have loaded.

13.3 "My pinned record disappeared"

Pinned records are local to your browser (localStorage). If you cleared site data, switched browsers, or are using a different device, you'll need to re-pin. Pinning is per-section: pinned campaigns don't show up under People.

13.4 "The landing page isn't live"

Check three things:

  1. Status is PUBLISHED (not DRAFT or ARCHIVED).
  2. Slug is set and unique.
  3. You're using the right URL: https://crm.geracillp.com/lp/<slug> — note the /lp/ prefix, not /landing-pages/.

13.5 "Email goes to spam"

Almost always a sender-auth issue. Verify:

  • SPF record on the sender domain includes the CRM's send infrastructure.
  • DKIM is configured and verified (Settings → Email sender domain).
  • The from-email matches the verified domain — no random gmail.com sender on a custom domain campaign.

13.6 "I broke something"

⌘ Z doesn't undo CRM operations. But you can:

  • Restore a deleted record from the trash (Settings → Data model → Trash).
  • Revert a campaign body — not currently versioned. Test sends before saving major edits.
  • Land in a weird state — sign out and back in. Most stuck UIs clear that way.
